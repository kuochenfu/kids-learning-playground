## Phase 1 — Security

### 1a. Environment variables for admin/parent emails

**Problem:** Admin email (`kuochenfu@gmail.com`) and parent email (`frankuo@gmail.com`) were hardcoded in the Go source.

**Fix:**
- Added `AdminEmail string`, `ParentEmails []string`, `CookieSecure bool` to `backend/config/config.go`
- `auth.go` and `main.go` now read these from env vars
- `.env.example` files created for both backend and frontend

**Files:** `backend/config/config.go`, `backend/services/auth.go`, `backend/main.go`, `backend/.env.example`, `frontend/.env.example`

---

### 1b. JWT in httpOnly cookie (XSS hardening)

**Problem:** JWT was stored in `localStorage` and sent as an `Authorization: Bearer` header — vulnerable to XSS token theft.

**Fix:**
- Backend login handler sets `jwt` as an httpOnly cookie (`SameSite=Lax` in dev, `SameSite=None;Secure` in production via `COOKIE_SECURE=true`)
- Added `POST /api/auth/logout` endpoint that clears the cookie server-side
- `AuthMiddleware` now reads from `c.Cookie("jwt")` instead of the Authorization header
- Frontend: created `frontend/src/utils/api.ts` — a centralized axios instance with `withCredentials: true`
- `AuthContext` now stores only the user profile in localStorage; no token in browser JS context
- All 10 game modules migrated from ad-hoc `axios + Authorization: Bearer` to the shared `api` utility

**Files:** `backend/main.go`, `frontend/src/utils/api.ts`, `frontend/src/context/AuthContext.tsx`, all game modules

---

### 1c. Rate limiting middleware

**Problem:** No rate limiting — login endpoint was open to brute-force.

**Fix:** In-process rate limiter using `sync.Map` + `time.Ticker`:
- 60 req/min for general routes
- 10 req/min on `POST /api/auth/google`
- Returns `429 Too Many Requests` with `Retry-After` header

**Files:** `backend/main.go`

---

### 1d. Request body validation

**Problem:** API handlers lacked explicit validation — malformed payloads could panic or produce silent errors.

**Fix:** Added Gin binding structs with `binding:"required"` tags to `/api/auth/google` and `/api/score` handlers.

## Bugs Found During Production Verification

After deploying and testing the live site at https://kids-learning-playground.pages.dev, three bugs were discovered and fixed:

### Bug 1 — Legacy `token` key not cleaned up on logout

**Root cause:** Old auth code stored the JWT as `localStorage.token`. New logout only removed `localStorage.user`.

**Fix:** Added `localStorage.removeItem('token')` to both the startup `useEffect` (cleans up existing sessions on first load) and the `logout()` function.

**Commit:** `fix: await logout before navigating and clean up legacy token key`

---

### Bug 2 — Logout didn't clear localStorage before navigation

**Root cause:** `handleLogout` in `MainLayout.tsx` called `logout()` without `await`, so `navigate('/login')` fired immediately while the async API call + localStorage cleanup was still in-flight.

**Fix:** Made `handleLogout` async and added `await logout()`.

```ts
// Before (race condition):
const handleLogout = () => { logout(); navigate('/login'); };

// After:
const handleLogout = async () => { await logout(); navigate('/login'); };
```

**Commit:** same as Bug 1

---

### Bug 3 — Role stayed `child` after adding email to `PARENT_EMAILS`

**Root cause:** The `PARENT_EMAILS` role check in `VerifyGoogleToken` only ran inside the `ErrRecordNotFound` branch (new user creation). Existing users were returned from the DB as-is — their role was never re-evaluated.

**Fix:** Moved role calculation outside the new/existing user conditional. On every login:
1. Derive `intendedRole` from `PARENT_EMAILS` env var
2. If new user → create with `intendedRole`
3. If existing user → update `Role`, `Name`, `Picture` in DB (also refreshes Google profile data)

**Benefit:** Role changes in `PARENT_EMAILS` now take effect on the user's very next login — no manual DB migration required.

**Commit:** `fix: re-evaluate user role on every login from PARENT_EMAILS config`

## Remaining Known Limitations

- **DB seeding strategy:** Drop-and-reseed on version mismatch is pragmatic but risky at scale — a proper migration tool (e.g., `golang-migrate`) would be safer.
- **Star Store:** Parent role is functional but the planned virtual reward/store feature is not yet built.
